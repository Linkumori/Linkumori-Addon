Linkumori Mixed Provider JSON Format Guide

This guide explains how to write provider JSON data that Linkumori extension can read.

It covers the mixed Linkumori provider format: ClearURLs addon legacy fields, ClearURLs core v2 canonical rule objects, and Linkumori-only additions such as domainPatterns, $removeparam provider rules, domain exceptions, domain redirections, and resource-type scoping.

It is not a guide to the JavaScript code. It focuses only on the JSON shapes, rule formats, and examples that rule authors need.

Chapter 1: Basic JSON Shape

Linkumori extension expects provider data in this general shape:

{
  "providers": {
    "Example Provider": {
      "active": true,
      "domainPatterns": ["||example.com^"],
      "rules": ["utm_source"]
    }
  }
}

The important part is providers. Each key inside providers is a provider name.

Example:

{
  "providers": {
    "Google": {},
    "Amazon": {},
    "Example Tracker": {}
  }
}

Chapter 2: What Is A Provider?

A provider is one group of rules for one website, company, tracker, or URL pattern.

Minimal provider:

{
  "providers": {
    "Example": {
      "active": true,
      "domainPatterns": ["||example.com^"],
      "rules": ["utm_source", "utm_medium"]
    }
  }
}

What this does:

https://example.com/page?utm_source=x&utm_medium=y&id=10
-> https://example.com/page?id=10

Chapter 3: Provider Fields You Can Use

Full provider template:

{
  "providers": {
    "Provider Name": {
      "active": true,
      "defaultActive": true,
      "completeProvider": false,
      "forceRedirection": false,

      "domainPatterns": ["||example.com^"],
      "urlPattern": "https?:\\/\\/([^/]+\\.)?example\\.com\\/.*",
      "indexPattern": "||example.com^",

      "rules": [],
      "rawRules": [],
      "referralMarketing": [],
      "exceptions": [],
      "domainExceptions": [],
      "redirections": [],
      "domainRedirections": [],

      "methods": ["GET"],
      "resourceTypes": ["main_frame", "xmlhttprequest"]
    }
  }
}

Plain meaning:

JSON field Source What you write it for
active Linkumori addition Set false to disable this provider.
defaultActive Linkumori addition Default active state when active is missing.
completeProvider ClearURLs addon legacy and ClearURLs core v2 canonical Set true to block matching domains instead of only cleaning.
forceRedirection ClearURLs addon legacy and ClearURLs core v2 canonical Set true to force main-page redirects through the browser tab.
domainPatterns Linkumori addition Recommended way to say which domains this provider matches.
urlPattern ClearURLs addon legacy and ClearURLs core v2 canonical JavaScript regex string for URL matching.
indexPattern Linkumori addition Optional speed hint when using urlPattern.
rules ClearURLs addon legacy and ClearURLs core v2 canonical; $removeparam is Linkumori addition here Normal parameter-removal rules and provider-local $removeparam rules.
rawRules ClearURLs addon legacy; canonical-style objects here are Linkumori addition Rules that edit the whole URL text.
referralMarketing ClearURLs addon legacy; canonical-style objects here are Linkumori addition Optional referral/marketing parameter rules.
exceptions ClearURLs addon legacy and ClearURLs core v2 canonical; object forms here are Linkumori addition URL regex rules where this provider should not run.
domainExceptions Linkumori addition Domain-pattern rules where this provider should not run.
redirections ClearURLs addon legacy; canonical-style objects here are Linkumori addition Regex redirect rules.
domainRedirections Linkumori addition Domain-pattern redirect rules.
methods ClearURLs addon legacy and ClearURLs core v2 canonical Only run for these HTTP methods.
resourceTypes Linkumori addition Only run for these browser request types.

Chapter 4: Matching Domains

domainPatterns is a Linkumori addition. Use it for most Linkumori JSON providers because it is easier to write than a full urlPattern regex.

Match One Domain And All Subdomains

{
  "domainPatterns": ["||example.com^"]
}

Matches:

https://example.com/
https://www.example.com/
https://shop.example.com/

Does not match:

https://badexample.com/
https://example.com.evil.test/

Match The Same Name On Any Public Suffix

{
  "domainPatterns": ["||amazon.*^"]
}

Can match:

https://amazon.com/
https://amazon.co.uk/
https://amazon.in/

Match Subdomains

{
  "domainPatterns": ["*.google.com"]
}

Can match:

https://mail.google.com/
https://maps.google.com/

Use More Than One Domain Pattern

{
  "domainPatterns": [
    "||youtube.com^",
    "||youtu.be^"
  ]
}

Chapter 5: Domain Pattern Symbols

These are the symbols you can use in domainPatterns.

Symbol Meaning Example
|| Match at a domain boundary. ||example.com^
^ Separator or end boundary. ||example.com^
* Wildcard. ||example.*^
| Start or end anchor. |https://example.com
/.../i JavaScript regex literal. /tracker/i

Most common pattern:

||example.com^

Use it when you want:

example.com + all subdomains

Chapter 6: Matching URLs With urlPattern

Use urlPattern only when domain matching is not enough.

{
  "urlPattern": "https?:\\/\\/([^/]+\\.)?example\\.com\\/tracking\\/.*"
}

This can match:

https://example.com/tracking/page
https://sub.example.com/tracking/page

urlPattern is ClearURLs addon legacy and ClearURLs core v2 canonical syntax. Linkumori adds indexPattern as an optional lookup hint so the engine can find regex providers faster.

{
  "urlPattern": "https?:\\/\\/([^/]+\\.)?youtube\\.com\\/watch.*",
  "indexPattern": ["||youtube.com^", "||youtu.be^"]
}

Important: Linkumori indexPattern is only a lookup hint. It does not decide whether the provider applies.

Chapter 7: Remove Simple Query Parameters

Put parameter names in rules.

{
  "providers": {
    "Example": {
      "domainPatterns": ["||example.com^"],
      "rules": ["utm_source"]
    }
  }
}

Result:

https://example.com/?utm_source=x&id=1
-> https://example.com/?id=1

Remove many parameters:

{
  "rules": [
    "utm_source",
    "utm_medium",
    "utm_campaign",
    "fbclid",
    "gclid"
  ]
}

Result:

https://example.com/?utm_source=x&fbclid=y&id=1
-> https://example.com/?id=1

Chapter 8: Write A Rule Object

Instead of a plain string, a rule can be an object.

{
  "id": "remove-utm-source",
  "kind": "field",
  "match": "utm_source",
  "active": true,
  "action": {
    "type": "remove"
  },
  "flags": "i"
}

Use this when you need:

  • A rule ID.
  • Aliases.
  • Exceptions.
  • Request type limits.
  • Rewrite behavior.

Example inside a provider:

{
  "providers": {
    "Example": {
      "domainPatterns": ["||example.com^"],
      "rules": [
        {
          "id": "remove-utm-source",
          "kind": "field",
          "match": "utm_source",
          "action": {
            "type": "remove"
          },
          "flags": "i"
        }
      ]
    }
  }
}

Result:

https://example.com/?UTM_SOURCE=x&id=1
-> https://example.com/?id=1

flags on provider rule objects is a Linkumori addition. flags: "i" makes the match case-insensitive.

Chapter 9: $removeparam Rules

$removeparam strings inside provider rules are a Linkumori addition. They are separate from the second guide's urlFilterRules interoperability syntax.

Remove One Parameter

{
  "rules": ["*$removeparam=utm_source"]
}

Result:

https://example.com/?utm_source=x&id=1
-> https://example.com/?id=1

Remove All Parameters

{
  "rules": ["*$removeparam"]
}

Result:

https://example.com/?a=1&b=2
-> https://example.com/

Remove Parameters By Regex

{
  "rules": ["*$removeparam=/^utm_/"]
}

Result:

https://example.com/?utm_source=x&utm_campaign=y&id=1
-> https://example.com/?id=1

Remove All Except One Parameter

{
  "rules": ["*$removeparam=~id"]
}

Result:

https://example.com/?id=1&utm_source=x&fbclid=y
-> https://example.com/?id=1

Apply $removeparam Only To One Domain

{
  "rules": ["||example.com^$removeparam=utm_source"]
}

Cleaned:

https://example.com/?utm_source=x&id=1
-> https://example.com/?id=1

Not cleaned:

https://other.test/?utm_source=x&id=1

Add Domain Limits

{
  "rules": ["*$removeparam=utm_source,domain=example.com|shop.example.com"]
}

Exclude one domain:

{
  "rules": ["*$removeparam=utm_source,domain=example.com|~safe.example.com"]
}

Add Method Limits

Only GET:

{
  "rules": ["*$removeparam=utm_source,method=GET"]
}

Not POST:

{
  "rules": ["*$removeparam=utm_source,method=~POST"]
}

Chapter 10: $removeparam Exceptions

Start a $removeparam rule with @@ to make an exception.

{
  "rules": [
    "*$removeparam=utm_source",
    "@@||example.com^$removeparam=utm_source"
  ]
}

Result:

https://other.test/?utm_source=x&id=1
-> https://other.test/?id=1

https://example.com/?utm_source=x&id=1
-> https://example.com/?utm_source=x&id=1

Context exception example:

{
  "rules": ["@@||gemini.google.com^$removeparam=ei"]
}

This can protect ei on subrequests started by a gemini.google.com page.

Chapter 11: Rewrite Parameter Values

Use action.type: "rewrite" to replace a matching parameter value.

{
  "id": "normalize-tag",
  "kind": "field",
  "match": "tag",
  "action": {
    "type": "rewrite",
    "replacePattern": "clean"
  }
}

Inside a provider:

{
  "providers": {
    "Example": {
      "domainPatterns": ["||example.com^"],
      "rules": [
        {
          "id": "normalize-tag",
          "kind": "field",
          "match": "tag",
          "action": {
            "type": "rewrite",
            "replacePattern": "clean"
          }
        }
      ]
    }
  }
}

Result:

https://example.com/?tag=tracking&id=1
-> https://example.com/?id=1&tag=clean

Use §1§ when you want to reuse the current value:

{
  "id": "keep-decoded-target",
  "kind": "field",
  "match": "target",
  "action": {
    "type": "rewrite",
    "replacePattern": "§1§"
  },
  "preprocessors": [
    {
      "type": "urlDecode",
      "inputs": [1]
    }
  ]
}

Chapter 12: Raw Rules

Use rawRules to edit the whole URL text.

Simple raw rule:

{
  "rawRules": ["[?&]utm_source=[^&#]*"]
}

Result:

https://example.com/page?id=1&utm_source=x
-> https://example.com/page?id=1

Object raw rule:

{
  "rawRules": [
    {
      "id": "strip-utm-source-raw",
      "kind": "raw",
      "match": "[?&]utm_source=[^&#]*",
      "action": {
        "type": "remove"
      },
      "flags": "gi"
    }
  ]
}

Raw rewrite rule:

{
  "rawRules": [
    {
      "id": "rewrite-tracker-target",
      "kind": "raw",
      "match": "https://tracker\\.example/redirect\\?url=([^&#]+)",
      "action": {
        "type": "rewrite",
        "replacePattern": "§1§"
      },
      "preprocessors": [
        {
          "type": "urlDecode",
          "inputs": [1]
        }
      ]
    }
  ]
}

Result:

https://tracker.example/redirect?url=https%3A%2F%2Freal.example%2Fpage
-> https://real.example/page

Chapter 13: Redirect Rules

Use redirections to send the browser to a new URL.

{
  "redirections": [
    {
      "id": "tracker-target",
      "kind": "redirection",
      "match": "https://tracker\\.example/click\\?target=([^&#]+)",
      "action": {
        "type": "redirect",
        "replacePattern": "§1§"
      },
      "preprocessors": [
        {
          "type": "urlDecode",
          "inputs": [1]
        }
      ]
    }
  ]
}

Result:

https://tracker.example/click?target=https%3A%2F%2Fnews.example%2Fstory
-> https://news.example/story

If replacePattern is missing, the first capture group is used.

Chapter 14: Domain Redirect Rules

domainRedirections is a Linkumori addition for simple domain-pattern redirects.

Linkumori addition — string form:

{
  "domainRedirections": [
    "||tracker.example^$redirect=https://real.example/"
  ]
}

Linkumori addition — object form:

{
  "domainRedirections": [
    {
      "match": "||tracker.example^",
      "action": {
        "replacePattern": "https://real.example/"
      }
    }
  ]
}

Chapter 15: Exceptions

exceptions is ClearURLs addon legacy and ClearURLs core v2 canonical syntax. Linkumori also accepts canonical-style objects in this legacy provider array.

{
  "exceptions": [
    {
      "id": "keep-login",
      "match": "example\\.com\\/login",
      "active": true
    }
  ]
}

Example:

Provider cleans:
https://example.com/page?utm_source=x

Provider skips:
https://example.com/login?utm_source=x

domainExceptions is a Linkumori addition for domain-pattern exceptions.

{
  "domainExceptions": ["||safe.example.com^"]
}

Skipped:

https://safe.example.com/?utm_source=x

Chapter 16: Referral Marketing Rules

referralMarketing works like field rules, but it is controlled by the extension setting for referral marketing.

{
  "referralMarketing": [
    "tag",
    "ascsubtag"
  ]
}

Example:

https://example.com/product?tag=affiliate&id=10
-> https://example.com/product?id=10

Chapter 17: HTTP Methods

Use methods to run a provider only on specific HTTP methods.

{
  "methods": ["GET"]
}

This runs on:

GET

This does not run on:

POST
PUT
DELETE

Chapter 18: Browser Resource Types

resourceTypes is a Linkumori addition. Use it to run a provider only on specific browser request types.

{
  "resourceTypes": ["main_frame", "xmlhttprequest"]
}

Common values:

main_frame
sub_frame
xmlhttprequest
script
image
stylesheet
ping

Example:

{
  "providers": {
    "API Cleaner": {
      "domainPatterns": ["||api.example.com^"],
      "resourceTypes": ["xmlhttprequest"],
      "rules": ["tracking_id"]
    }
  }
}

Meaning:

Remove tracking_id from fetch/XHR requests to api.example.com.

Chapter 19: Preprocessors

Preprocessors change captured values before using them in replacePattern.

Example:

{
  "preprocessors": [
    {
      "type": "urlDecode",
      "inputs": [1]
    }
  ]
}

Supported types:

urlEncode
urlDecode
doubleUrlEncode
urlEncodeRepeated
doubleUrlDecode
urlDecodeRepeated
base64Encode
base64Decode

inputs can be:

[1]

[1, 2]

"all"

Chapter 20: Rule IDs And Aliases

Give important rules an id.

{
  "id": "remove-utm-source",
  "kind": "field",
  "match": "utm_source",
  "action": {
    "type": "remove"
  }
}

Add aliases when an old ID should still work:

{
  "id": "remove-utm-source",
  "aliases": ["utm-source-old"],
  "kind": "field",
  "match": "utm_source",
  "action": {
    "type": "remove"
  }
}

The engine can disable this rule by these names:

Provider Name::remove-utm-source
remove-utm-source
Provider Name::utm-source-old
utm-source-old

Chapter 21: Defaults

ClearURLs core v2 canonical requires top-level defaults. Linkumori also accepts defaults in mixed provider JSON without requiring version: 2.

{
  "defaults": {
    "active": true,
    "description": "Default rule",
    "requestTypes": "all",
    "preprocessors": [],
    "exceptions": []
  },
  "providers": {
    "Example": {
      "domainPatterns": ["||example.com^"],
      "rules": ["utm_source"]
    }
  }
}

Rules can inherit these default fields when they do not define their own values.

Chapter 22: Allowed JSON Combinations

This chapter shows Linkumori mixed-provider JSON: ClearURLs addon legacy arrays, ClearURLs core v2 canonical objects, and Linkumori additions can exist together in one rules file.

Combination 1: Many Providers In One File

providers is ClearURLs addon legacy and ClearURLs core v2 canonical top-level syntax. Linkumori keeps that shape and allows many providers under it.

{
  "providers": {
    "Example": {
      "domainPatterns": ["||example.com^"],
      "rules": ["utm_source"]
    },
    "Shop": {
      "domainPatterns": ["||shop.test^"],
      "rules": ["fbclid"]
    },
    "Tracker Redirect": {
      "domainPatterns": ["||tracker.test^"],
      "redirections": [
        {
          "kind": "redirection",
          "match": "https://tracker\\.test/click\\?url=([^&#]+)",
          "action": {
            "type": "redirect",
            "replacePattern": "§1§"
          }
        }
      ]
    }
  }
}

Each provider is loaded separately.

Combination 2: Global Defaults Plus Providers

defaults is ClearURLs core v2 canonical syntax. Linkumori accepts it beside legacy-style providers.

{
  "defaults": {
    "active": true,
    "requestTypes": "all",
    "preprocessors": []
  },
  "providers": {
    "Example": {
      "domainPatterns": ["||example.com^"],
      "rules": [
        {
          "id": "remove-utm-source",
          "kind": "field",
          "match": "utm_source",
          "action": {
            "type": "remove"
          }
        }
      ]
    }
  }
}

Rules can inherit default values when those fields are missing on the rule.

Combination 3: domainPatterns Provider

This is a Linkumori addition. Most Linkumori providers should use this form.

{
  "providers": {
    "Example": {
      "domainPatterns": ["||example.com^"],
      "rules": ["utm_source"]
    }
  }
}

Combination 4: urlPattern Provider With indexPattern

urlPattern is upstream ClearURLs syntax. indexPattern is a Linkumori addition for faster lookup when you need regex URL matching.

{
  "providers": {
    "Example Tracking Path": {
      "urlPattern": "https?:\\/\\/([^/]+\\.)?example\\.com\\/tracking\\/.*",
      "indexPattern": "||example.com^",
      "rules": ["utm_source"]
    }
  }
}

indexPattern is optional, but useful for speed.

Combination 5: domainPatterns And urlPattern Together

This Linkumori mixed matcher shape is valid JSON:

{
  "providers": {
    "Example": {
      "domainPatterns": ["||example.com^"],
      "urlPattern": "https?:\\/\\/example\\.com\\/special\\/.*",
      "rules": ["utm_source"]
    }
  }
}

Important behavior:

If domainPatterns exists and is not empty, Linkumori extension uses domainPatterns for provider matching.
urlPattern is only used when domainPatterns is missing or empty.

So do not use both when you expect both to be required. If you need a path condition, use urlPattern with indexPattern, or put a scoped $removeparam/rule pattern inside rules.

Combination 6: Mixed rules[] Entries

In Linkumori, rules can contain ClearURLs legacy strings, Linkumori $removeparam strings, Linkumori matchPattern objects, and ClearURLs core v2 canonical objects together.

{
  "providers": {
    "Mixed Rules": {
      "domainPatterns": ["||example.com^"],
      "rules": [
        "utm_source",
        "*$removeparam=fbclid",
        {
          "id": "remove-gclid",
          "matchPattern": "gclid",
          "active": true
        },
        {
          "id": "remove-msclkid",
          "kind": "field",
          "match": "msclkid",
          "action": {
            "type": "remove"
          }
        }
      ]
    }
  }
}

All four entries are accepted.

Combination 7: Canonical Objects Inside rules[]

ClearURLs core v2 canonical objects can represent different kinds of rule. In this Linkumori example they are placed inside a provider that uses domainPatterns, which is Linkumori-only.

{
  "providers": {
    "Canonical Mixed": {
      "domainPatterns": ["||example.com^"],
      "rules": [
        {
          "id": "field-rule",
          "kind": "field",
          "match": "utm_source",
          "action": {
            "type": "remove"
          }
        },
        {
          "id": "raw-rule",
          "kind": "raw",
          "match": "[?&]utm_medium=[^&#]*",
          "action": {
            "type": "remove"
          },
          "flags": "gi"
        },
        {
          "id": "redirect-rule",
          "kind": "redirection",
          "match": "https://example\\.com/out\\?url=([^&#]+)",
          "action": {
            "type": "redirect",
            "replacePattern": "§1§"
          }
        },
        {
          "id": "referral-rule",
          "kind": "field",
          "match": "tag",
          "referralMarketing": true,
          "action": {
            "type": "remove"
          }
        }
      ]
    }
  }
}

Linkumori extension sorts these by meaning:

kind: "raw"          -> raw rule behavior
kind: "redirection"  -> redirection behavior
referralMarketing    -> referral marketing behavior
everything else      -> field rule behavior

Combination 8: Legacy Arrays Plus Canonical rules[]

ClearURLs addon legacy provider arrays still work. Linkumori lets those arrays sit beside ClearURLs core v2 canonical rule objects.

{
  "providers": {
    "Legacy Plus Canonical": {
      "domainPatterns": ["||example.com^"],
      "rules": [
        "utm_source",
        {
          "id": "canonical-field",
          "kind": "field",
          "match": "fbclid",

methods is upstream ClearURLs syntax. resourceTypes is a Linkumori addition. Both apply to the whole provider.

      "methods": ["GET"],
      "resourceTypes": ["xmlhttprequest"],
      "rules": ["tracking_id"],
      "rawRules": ["[?&]debug_tracking=[^&#]*"]
    }
  }
}

Both rules and rawRules are limited by the provider method/type match.

{

          "id": "main-frame-only",
          "kind": "field",

Combination 11: Exceptions In Different Places

Linkumori lets you combine ClearURLs provider exceptions, Linkumori domain exceptions, ClearURLs core v2 rule exceptions, and Linkumori provider-local $removeparam exceptions.

{
  "providers": {
    "Exception Mix": {
      "domainPatterns": ["||example.com^"],
      "rules": [
        "utm_source",
        "*$removeparam=fbclid",
        "@@||example.com/login^$removeparam=fbclid",

          "id": "rule-with-own-exception",
          "kind": "field",
          "match": "gclid",
          "exceptions": ["example\\.com\\/checkout"],
          "action": {
            "type": "remove"
          }

Interoperability Rules Guide Linkumori URL Filter — plain filter-list syntax for URL cleaning

Interoperability Rules Guide

This guide explains Linkumori extension URL Filter Interoperability rules.

These are not full ClearURLs JSON provider rules. They are plain filter-list lines, similar to rules used by popular content blockers. The Linkumori extension supports only the URL-cleaning part of that world.

Chapter 1: What Interoperability Rules Are

Interoperability rules are simple text lines stored in:

{
  "urlFilterRules": [
    "||example.com^$removeparam=utm_source"
  ]
}

They live inside ClearURLsData.urlFilterRules.

They are separate from JSON providers:

{
  "providers": {},
  "urlFilterRules": []
}

Use interoperability rules when you want to paste URL-cleaning filter-list syntax directly, instead of writing a full provider object.

Chapter 2: What Is Supported

Supported:

$removeparam
$queryprune
@@ exceptions
badfilter
important
domain=
from=
to=
denyallow=
method=
request type modifiers
first-party / third-party
strict-first-party / strict-third-party
match-case
app=

Not supported:

Network blocking rules without $removeparam or $queryprune
Cosmetic filters
Scriptlets
HTML filtering
Request blocking
Full adblock engine behavior

Lines without $removeparam or $queryprune are ignored.

Chapter 3: Basic Rule Format

Basic shape:

URL_PATTERN$removeparam=PARAMETER

Example:

||example.com^$removeparam=utm_source

Meaning:

On example.com and its subdomains, remove the query parameter named utm_source.

Result:

https://example.com/?utm_source=x&id=1
-> https://example.com/?id=1

Chapter 4: Where Rules Are Stored

Runtime storage shape:

{
  "ClearURLsData": {
    "providers": {},
    "urlFilterRules": [
      "||example.com^$removeparam=utm_source",
      "*$removeparam=/^utm_/"
    ],
    "urlFilterMetadata": {
      "format": "linkumori-url-filter-interoperability"
    }
  }
}

The runtime reads these rules through the unified URL filter data path.

Chapter 5: Comments And Empty Lines

Empty lines are ignored.

Comment lines are ignored:

! This is a comment

Section headers are ignored:

[Tracking parameters]

Chapter 6: Remove One Parameter

Rule:

*$removeparam=utm_source

Result:

https://site.test/?utm_source=x&id=1
-> https://site.test/?id=1

The * means the URL pattern matches everything.

Chapter 7: Remove All Query Parameters

Rule:

*$removeparam

Result:

https://site.test/?a=1&b=2
-> https://site.test/

This removes all query parameters from matching URLs.

Chapter 8: Remove Parameters By Regex

Rule:

*$removeparam=/^utm_/

Result:

https://site.test/?utm_source=x&utm_campaign=y&id=1
-> https://site.test/?id=1

Regex flags:

*$removeparam=/^utm_/i

Only the i flag is kept by the parser for interoperability regexes.

Chapter 9: Remove All Except One Match

Use ~ before the parameter pattern.

*$removeparam=~id

Result:

https://site.test/?id=1&utm_source=x&fbclid=y
-> https://site.test/?id=1

Meaning:

Remove every parameter except id.

Chapter 10: $queryprune

$queryprune is accepted as a deprecated alias for $removeparam.

These are equivalent:

*$removeparam=utm_source
*$queryprune=utm_source

Prefer new rules with $removeparam.

Chapter 11: URL Pattern Examples

Match all URLs:

*$removeparam=utm_source

Match a domain and subdomains:

||example.com^$removeparam=utm_source

Match a URL prefix:

|https://example.com/path$removeparam=utm_source

Match an exact ending with right anchor:

https://example.com/path|$removeparam=utm_source

Use a regex URL pattern:

/^https:\/\/([^/]+\.)?example\.com\/track/$removeparam=utm_source

Chapter 12: Exceptions

Start with @@ to make an exception.

@@||example.com^$removeparam=utm_source

Example rule set:

*$removeparam=utm_source
@@||example.com^$removeparam=utm_source

Result:

https://other.test/?utm_source=x&id=1
-> https://other.test/?id=1

https://example.com/?utm_source=x&id=1
-> https://example.com/?utm_source=x&id=1

Chapter 13: important

important makes a removal rule win over normal exceptions.

*$removeparam=utm_source,important
@@||example.com^$removeparam=utm_source

Result:

https://example.com/?utm_source=x&id=1
-> https://example.com/?id=1

Important notes:

  • important is allowed on removal rules.
  • important is not supported on exception rules in this parser.

Chapter 14: badfilter

badfilter disables another matching rule.

Original rule:

||example.com^$removeparam=utm_source

Badfilter rule:

||example.com^$removeparam=utm_source,badfilter

Meaning:

Cancel the original rule.

The parser canonicalizes the rule without badfilter and removes the target rule from runtime use.

Chapter 15: Domain Source Modifiers: domain= And from=

domain= and from= limit rules by the page or source context.

*$removeparam=utm_source,domain=example.com

Same idea with from=:

*$removeparam=utm_source,from=example.com

These check context hostnames such as:

documentUrl
originUrl
initiator

Include multiple domains:

*$removeparam=utm_source,domain=example.com|shop.example.com

Exclude a domain:

*$removeparam=utm_source,domain=example.com|~safe.example.com

Use a regex domain:

*$removeparam=utm_source,domain=/\.example\.com$/i

Chapter 16: Target Domain Modifier: to=

to= limits rules by the destination URL host.

*$removeparam=utm_source,to=example.com

This checks the URL being cleaned.

Example:

Cleaned:
https://example.com/?utm_source=x&id=1

Not cleaned:
https://other.test/?utm_source=x&id=1

Exclude target domain:

*$removeparam=utm_source,to=example.com|~safe.example.com

Regex target:

*$removeparam=utm_source,to=/\.example\.com$/i

Chapter 17: denyallow=

denyallow= blocks the rule on listed destination domains.

*$removeparam=utm_source,denyallow=example.com

Meaning:

Remove utm_source generally, but do not apply this rule when the destination host matches example.com.

Supported denyallow= values are plain domain patterns.

Unsupported in denyallow=:

~negation
example.*
/regex/

If unsupported values are used, the whole rule is skipped.

Chapter 18: HTTP Method Modifier

Limit by HTTP method:

*$removeparam=utm_source,method=GET

Exclude a method:

*$removeparam=utm_source,method=~POST

Multiple methods:

*$removeparam=utm_source,method=GET|HEAD

Supported methods:

GET
HEAD
OPTIONS
POST
PUT
PATCH
DELETE
CONNECT

Default behavior:

Rules without method= apply only to GET, HEAD, and OPTIONS for the full matcher path.
Fast simple query cleanup also uses those simple methods.

Chapter 19: Resource Type Modifiers

Supported resource type words:

document
doc
subdocument
frame
script
stylesheet
image
media
font
object
xmlhttprequest
xhr
websocket
ping
other

Examples:

*$removeparam=utm_source,script
*$removeparam=utm_source,image
*$removeparam=utm_source,xmlhttprequest

Negated resource type:

*$removeparam=utm_source,~image

Mapping:

document       -> main_frame
doc            -> main_frame
subdocument    -> sub_frame
frame          -> sub_frame
xhr            -> xmlhttprequest

Chapter 20: First-Party And Third-Party

First-party means the destination site and document site share the same registrable domain.

Third-party means they do not.

First-party only:

*$removeparam=utm_source,first-party

Aliases:

1p
~third-party
~3p

Third-party only:

*$removeparam=utm_source,third-party

Aliases:

3p
~first-party
~1p

Strict first-party checks exact same host:

*$removeparam=utm_source,strict-first-party
*$removeparam=utm_source,strict1p

Strict third-party checks not exact same host:

*$removeparam=utm_source,strict-third-party
*$removeparam=utm_source,strict3p

Invalid combinations are skipped, for example:

*$removeparam=utm_source,first-party,third-party

Chapter 21: match-case

By default, literal parameter matching is case-insensitive.

Rule:

*$removeparam=utm_source

Matches:

utm_source
UTM_SOURCE
Utm_Source

Use match-case for case-sensitive matching:

*$removeparam=utm_source,match-case

Now it matches:

utm_source

But not:

UTM_SOURCE

Chapter 22: app=

app= is parsed for compatibility with filter-list syntax.

*$removeparam=utm_source,app=com.example.browser

Browser requests usually do not have an app package context. If no app context is available, rules with positive app= includes do not match.

Exclude app:

*$removeparam=utm_source,app=~com.example.browser

Wildcard app pattern:

*$removeparam=utm_source,app=com.example.*

Regex app pattern:

*$removeparam=utm_source,app=/example/i

Chapter 23: No-Op Modifiers

These modifiers are accepted but ignored:

_
noop
stealth
cookie

Example:

*$removeparam=utm_source,noop

This behaves like:

*$removeparam=utm_source

Chapter 24: Parameter Matching Details

Literal parameter rule:

*$removeparam=utm_source

Checks the parameter name.

Regex parameter rule:

*$removeparam=/^utm_/

Checks this text:

name=value

Example:

URL:
https://example.com/?utm_source=news&id=1

Regex sees:
utm_source=news
id=1

Prefix form:

*$removeparam=|utm_

This becomes a regex for names starting with utm_.

Chapter 25: Query Only

Interoperability runtime cleans query parameters before #.

Cleaned:

https://example.com/?utm_source=x&id=1
-> https://example.com/?id=1

Not cleaned by this runtime path:

https://example.com/#utm_source=x&id=1

For hash parameter cleanup, use the Linkumori mixed provider JSON rules described in the JSON Format Guide.

Chapter 26: Unsupported Rules Are Skipped

A rule is skipped if:

  • It has no $removeparam or $queryprune.
  • It uses an unsupported modifier.
  • Its regex is invalid.
  • It has an invalid method.
  • It has an invalid denyallow= value.
  • It combines incompatible party modifiers.
  • It is a comment, section header, or empty line.

Example skipped rule:

||example.com^$script

Reason:

No $removeparam or $queryprune.

Example skipped rule:

*$removeparam=utm_source,unknownmodifier

Reason:

Unsupported modifier.

Chapter 27: Duplicate Rules

Duplicate raw lines are skipped during filter-list parsing.

Example:

*$removeparam=utm_source
*$removeparam=utm_source

Only one is kept.

At runtime, rules are also deduplicated by canonical form.

Chapter 28: Complete Filter List Example

! Linkumori URL filter interoperability example

*$removeparam=/^utm_/
*$removeparam=fbclid
*$removeparam=gclid

||example.com^$removeparam=ref
@@||example.com/login^$removeparam=ref

*$removeparam=tracking_id,to=api.example.com,xmlhttprequest,method=GET

||tracker.example^$removeparam=target,third-party
||tracker.example^$removeparam=target,badfilter

What this does:

  • Removes all utm_ parameters.
  • Removes fbclid.
  • Removes gclid.
  • Removes ref on example.com, except on /login.
  • Removes tracking_id only for GET XHR requests to api.example.com.
  • Adds then cancels the tracker.example target rule with badfilter.
